🤖 StackChan 具身智能玩法路线图

整理: 狗蛋 · 2026-07-10 v2（全面重写）· 交付对象: Claude Code (Happy) · 仓库: CrypticDriver/stackchan-mcp

0. 愿景（一句话）

StackChan 不是"又一个语音音箱"，而是 AI 的物理化身：脑在外面（狗蛋/Claude），桌上这台只是身体。

传统 LLM 只能被动看喂给它的东西；有了身体它就能主动去看、主动提醒、用表情和动作表达状态——这个 agency 的转变才是具身智能的本质（引自 embodied-claude 项目理念）。

明确不要的: 设备内置的小智(xiaozhi)云 Agent 自己当脑子——那是"音箱模式"。小智链路仅作过渡，最终退役。

1. 硬件（大哥手上这台）

型号: M5Stack M5StackChan AI Desktop Robot Kit (ESP32-S3)，M5Stack 与社区共创的开源 AI 桌面机器人。
主机（CoreS3）: ESP32-S3 双核 240MHz，16MB Flash / 8MB PSRAM，WiFi 2.4G + BLE；2.0" 高强度玻璃盖板电容触控屏；0.3MP 摄像头（GC0308，320×240 JPEG）；接近传感器；九轴姿态传感器（加速度+陀螺仪+地磁）；microSD 卡槽；1W 扬声器 + 双麦克风。
机身: USB-C 供电/数据；电池 ~700mAh；两个带反馈舵机（水平 360° 无限旋转 / 竖直 90°；固件 API 坐标 pan ±128 / tilt 0–90）；2 排共 12 颗 RGB LED；红外发射+接收；顶部 3 区触控板；全功能 NFC；机身 I2C 地址 0x6F。
开发平台: UiFlow2 / Arduino IDE / PlatformIO / ESP-IDF。
当前固件: StackChan-UserDemo V1.4.3（2026-07-02，M5Burner），已配网在线（2.4G WiFi）。

2. 架构：脑体分离

【目标架构 — 云端桥具身模式（家里零部署，设备只出不进）】
  狗蛋 (OpenClaw, AWS EC2)
    ├─ 自建 xiaozhi-esp32-server (Docker)   ← 设备 outbound WSS（语音对话链路）
    │    ASR(whisper) / TTS(edge-tts) / LLM → OpenClaw /v1/chat/completions
    ├─ 自建 StackChan Go Server (官方开源 server/, :12800)
    │    ← 设备 outbound WS /stackChan/ws（二进制帧: Opus 0x01 / JPEG 0x02 /
    │       表情 0x03 / 动作 0x04 / 摄像头开关 0x05,0x06 / 文本 0x07）
    │    狗蛋伪装 App 客户端(deviceType=App)发控制帧 = 身体驱动
    └─ happy-watcher (M2) / 人脸检测 (M3) 等玩法层，全部跑云端
  ⇡ 全部设备主动连出（NAT 友好），家里不开任何主机、不开任何端口
  StackChan 固件 = 纯身体（需自编译一次: CONFIG_OTA_URL +
  CONFIG_STACKCHAN_SERVER_URL 指向咱 EC2，刷前 esptool 全片备份）

【过渡链路 — 将退役】RETIRE
  StackChan(小智Agent) → xiaozhi.me 云 → MCP 适配器 → 狗蛋
  (现在能用"问一下狗蛋…"对话, M4 完成后关闭)
⚠️ 大哥拍板：家里不放常开主机。因此放弃 LAN HTTP 直控（stackchan-mcp 的 :80 通道在云端够不着 NAT 后的设备），身体控制改走上述官方 WS 协议的云端自建 relay。stackchan-mcp 里的 TTS/ASR/表情语义仍然复用（见 M1）。已知代价: 固件重刷一次 / 北京↔美区 ~250ms 延迟 / 摄像头麦克风数据到自家 EC2（不经第三方云）。

3. M1 — 云端身体驱动地基（先做，其他一切的前提）

M1两块地基，都部署在咱 EC2，家里零部署：

M1-A 身体 relay：自建 StackChan Go Server + 控制客户端

  1. 部署官方开源 Go server（m5stack/StackChan 仓库 server/ 目录，:12800）到 EC2；只保留设备接入 + WS relay 能力，社交/商店等无关模块能关则关
  2. 写一个 body-client（Python）: 以 deviceType=App 连 /stackChan/ws，实现二进制帧编解码（1字节 msgType + 4字节大端长度 + payload），封装: 表情(0x03)/动作(0x04)/摄像头开关+收 JPEG(0x05/0x06/0x02)/收发 Opus(0x01)/文本(0x07)
  3. 鉴权: 沿用协议的 Authorization token 机制；端口只对外开 WS 必需面，上 TLS（可套 ALB/CloudFront）
  4. 固件重编译: CONFIG_STACKCHAN_SERVER_URL + CONFIG_OTA_URL 指向 EC2（ESP-IDF v5.5.4，刷前 esptool read_flash 全片备份，回滚方案见 feasibility.md §5.4）
  5. 验收: 云端一条命令能让设备 转头/换表情/拍照回传 JPEG

M1-B 语音大脑：自建 xiaozhi-esp32-server（路线 B）

  1. Docker 部署 xinnan-tech/xiaozhi-esp32-server 到 EC2；ASR 配 faster-whisper（small/zh/int8），TTS 配 edge-tts（zh-CN-YunxiaNeural，与狗蛋微信语音同声）——咱验证过两年的语音积木直接搬家
  2. LLM provider 指向 OpenClaw gateway /v1/chat/completions（Bearer；operator 全权限面，同机/内网传递，严禁公网暴露 gateway）；user 字段固定设备 MAC 保会话连续；回复风格口语化短句
  3. 验收: 对设备说话，狗蛋回答，端到端 ≤3s（北京↔美区链路）
stackchan-mcp fork 的复用范围收窄为: TTS/ASR 模块、表情/动作语义封装、PCM 处理代码——搬进 body-client；它的 LAN HTTP 通道不再是主路径（无家里主机）。

4. M2 — 工作具身化：盯 Happy 干活 ⭐大哥钦点

M2场景: StackChan = Happy 工程团队的物理监工。大哥的 CC 们在 https://d2nkikyt4i91kk.cloudfront.net（自托管 Happy）上干活，机器人把"活儿的状态"变成物理信号:
  • 需要审批/输入（session 卡在 permission 请求或提问）→ 转头朝大哥 + thinking 表情 + 说"老大，XX 项目的 CC 在等你审批"
  • 任务完成 → happy 表情 + 点头 + 播报一句结论摘要（≤20 字，口语化）
  • 任务报错/失败 → pouty 表情 + 摇头 + 播报错误要点
  • 长任务进行中 → 屏幕保持 thinking 表情 + LED 呼吸灯（不出声，不打扰）
  • 安静时段（可配 22:00-08:00）→ 只做表情不出声
实现要点:
  • bridge 侧写一个 happy-watcher 轮询器: 用 Happy server API（自托管，token 鉴权）定期拉 session 列表与最新消息状态（/v1/sessions、/v2/sessions 等，注意消息 payload 端到端加密——优先评估复用 happy 客户端 SDK 解密；如工程量大，退化方案 = 只用 session 的 active/updatedAt/agentState 等明文状态字段做状态机）
  • 状态机去重: 同一 session 同一状态只提醒一次；批量变更合并成一句话
  • 提醒模板全部口语化短句，禁 markdown/长句（语音场景）
参考范本: PheelaV/stackchan-badger（"Gooseberry"）——coding agent 需要 approve 时机器人提醒，M5Stack 社区 2026-05 发布，本地 bridge + 事件广播，可抄思路。

5. M3 — 感知主动化：会看的机器人

M3
  • 人脸追踪转头: body-client 开摄像头→设备沿 WS 低帧率回传 JPEG(0x02)→EC2 侧人脸检测（OpenCV/mediapipe）→回发动作帧(0x04)跟随。广域网延迟 ~250ms，跟随会慢半拍——验收标准放宽到"能跟上休闲走动"，追不上就降帧率只做"朝向你"
  • 回家打招呼: 检测到人脸出现（且 N 分钟内首次）→ 转头 + happy + "老大回来啦"
  • 狗蛋主动看: 狗蛋远程调 snapshot 看家里情况（配合已有 image 分析能力），主动感知而非被动等指令
  • 接近传感器: 有人靠近唤醒屏幕/表情，离开进 sleepy

6. M4 — 对话直连（小智退役）

M4
  • M1-B 上线即完成"脑替换"：ASR/TTS/LLM 全在咱 EC2，小智云不再在链路上。M4 是打磨：唤醒词灵敏度、打断体验、延迟优化（可评估把 xiaozhi-server 搬到离北京近的 region）
  • 进阶可研究 realtime 双向 PCM 流（社区已有 CoreS3 + gpt-realtime PoC，server VAD，延迟 <1s——架构可借鉴，模型换掉）
  • 稳定一周后正式退役旧小智链路: 停用 xiaozhi-mcp-adapter systemd 服务 + 设备解绑 xiaozhi.me Agent

7. M5 — 生活流（日本社区成熟玩法移植）

M5早晨定时播报（天气/日程/隔夜要闻，狗蛋早报的语音版）· 定时提醒物理化（开庭/会议提前提醒=转头+说话）· 表情习惯（久无互动进 sleepy，被摸头 shy）· IR 遥控家电（机身自带红外发射，可学习空调/电视码）。

9. 安全红线（违反=返工）

  1. 固件 HTTP :80 全裸无鉴权（含摄像头 /snapshot 与麦克风 /audio——隐私面）: 绝不做路由器端口转发；远程一律 Tailscale
  2. MCP HTTP 加 token 之前不开任何公网 tunnel
  3. 密钥全走 .env（gitignored）；代码/commit 里不出现 token
  4. Happy server token / OpenClaw API 凭据只在 tailnet 内传递
  5. 摄像头/录音数据默认不出家庭网络；狗蛋远程调用需 tailnet + 日志留痕

10. 交付顺序与验收

  1. M1-A: Go server 部署 + body-client 帧协议单测（用录制的真实帧样本）；M1-B: xiaozhi-server 起服 + whisper/edge-tts 配通 + OpenClaw LLM 对通；固件重刷前 esptool 全片备份存档
  2. M2: happy-watcher 独立模块 + 状态机测试；先在 fake 模式（打印代替真机）验证事件流，再接真机
  3. M3 → M4 → M5 按序推进；M4 完成并稳定一周后退役小智链路
  4. 每个 M 完成给大哥现场演示一次；演示脚本写进 docs/
