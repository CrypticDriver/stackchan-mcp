1. 背景与定位

设备型号: M5Stack M5StackChan AI Desktop Robot Kit (ESP32-S3) — 主控 CoreS3（ESP32-S3 双核 240MHz，16MB Flash / 8MB PSRAM，WiFi 2.4G + BLE），双伺服云台（pan ±128 / tilt 0–90），2.0" 电容触屏，GC0308 摄像头（0.3MP，320×240 JPEG），双麦克风 + 扬声器，NFC；机身 I2C 地址 0x6F。
固件: StackChan-UserDemo V1.4.3（2026-07-02 版，M5Burner 烧录），已配网在线（2.4G WiFi）。
定位（大哥拍板）: 家里的桌面管家 — 陪伴 / 语音入口 / 家庭中控。不做随身感知（那是智能眼镜的活）。
终局目标: StackChan = 狗蛋(OpenClaw 主 Agent)的物理化身：能听（叫狗蛋办事）、能说（狗蛋的语音回复）、能动（点头/摇头/转头）、能看（拍照给狗蛋分析）、有表情。

3. 两条链路的架构（重要，别搞混）

链路① 语音对话（已通，云端反向连接，无需家里开端口）:
  StackChan 固件(xiaozhi-esp32 v2.2.4 组件)
    --Opus/WSS--> xiaozhi.me 云
    --MCP wss(JWT)--> 咱的适配器(EC2, systemd)
    --SigV4--> AgentCore MCP runtime --> OpenClaw gateway --> 狗蛋

链路② 本地精细控制（stackchan-mcp，待增强 = 本任务书主体）:
  控制端(狗蛋/Claude) --> MCP server(Python, :8002)
    --HTTP/TCP/UDP(LAN)--> 固件 HTTP :80
  能力: TTS 播放 / 录音转写 / 舵机动作 / 7 种表情 / 摄像头抓拍
链路①管"对话"，链路②管"身体"。链路②的 MCP server 需要跑在与设备同 LAN 的常开主机上（树莓派/N100/Mac），远程接入靠 Tailscale。

4. 本次任务：stackchan-mcp fork 增强（Phase 2 快赢包）

仓库: github.com/CrypticDriver/stackchan-mcp（fork 自 migratorywhale/stackchan-mcp @ d90c870）。Python 3.11+ / FastMCP，约 2500 行。以下按优先级排序，全部只改 Python 不动固件。

P0ASR 可插拔 + faster-whisper 本地后端

  • 痛点: stackchan_listen 硬依赖 Fish Audio API key（audio_processing.py L281 transcribe_audio；listening.py L27 无 key 直接抛错）——咱没有 Fish key，"听"完全不可用
  • 做法: 抽象 ASR_ENGINE=fish|whisper 环境变量；whisper 后端用 faster-whisper（small 模型, zh, CPU int8, beam_size=5），参考实现已验证: 
    from faster_whisper import WhisperModel
model = WhisperModel("small", device="cpu", compute_type="int8")
segments, info = model.transcribe(wav_path, language="zh", beam_size=5)
text = " ".join(seg.text for seg in segments)
  • 验收: 无任何 Fish key 时 listen 工具端到端可用，中文识别正确

P0MCP HTTP 模式加 Bearer token 鉴权

  • 痛点: FastMCP streamable-http (:8002) 无任何 auth，远程接入前置条件
  • 做法: ASGI middleware 校验 Authorization: Bearer $STACKCHAN_MCP_TOKEN；未配置 token 时保持现状（本地 stdio 不受影响）；401 响应不泄漏细节
  • 验收: 无 token 401；带错 token 401；带对 token 全工具正常

P1edge-tts 声音可配置

  • EDGE_VOICES 硬编码 zh=YunxiNeural（stackchan_config.py L146）→ 加 STACKCHAN_EDGE_VOICE_ZH 环境变量，默认保持原值；咱要用 zh-CN-YunxiaNeural（卡通男声，与狗蛋微信语音一致）

P1TTS 结果缓存

  • hash(文本+voice+engine) → WAV 文件缓存（LRU，上限可配如 200MB），重复短语（问候/口头禅）零延迟、零 API 调用

P2PCM 处理 numpy 向量化

  • condition_pcm_chunk / declick / zero-cross（audio_processing.py L196-249）是纯 Python 逐 sample 循环，树莓派上长语音会吃满 CPU → numpy 化，预期 ~10x
  • 验收: 输出与原实现逐字节一致（写对比测试），树莓派/低配机 CPU 占用显著下降

P2（可选）中等难度固件项——先别做，等大哥确认

  • 固件 HTTP 共享 token（config.h + http_server.cpp 每 handler 校验 X-Stackchan-Token，~50 行，需重刷固件）
  • main loop 看门狗 + heap 低水位自动重启；loop() delay(50) 调优；VAD 参数化
  • 动固件必须先 esptool read_flash 全片备份（详见 §6）

5. 第二阶段（独立任务，本次不做）: 自建 xiaozhi-esp32-server

  • 目标: 语音对话链路脱离 xiaozhi.me 云，全自托管——部署 xinnan-tech/xiaozhi-esp32-server（Docker），LLM provider 用 openai 类型指向 OpenClaw gateway /v1/chat/completions（Bearer token；该端点是 operator 全权限面，只能内网/tailnet，严禁公网）
  • 固件把 CONFIG_OTA_URL 指向自建 server（ESP-IDF v5.5.4 自编译 + 刷机，刷前全片备份）
  • OpenClaw 侧: model 字段填 openclaw/<agentId>；user 字段固定为设备 MAC 保持会话连续；agent 输出需口语化短句（语音场景）
  • 工作量 1-2 人日；先拿测试设备验证再动主力机

6. 安全红线（违反=返工）

  1. 固件 HTTP :80 全裸无鉴权（含摄像头 /snapshot 和麦克风 /audio——隐私面）：绝不做路由器端口转发；远程一律 Tailscale
  2. MCP HTTP 加 token 之前不开 cloudflared tunnel
  3. 所有密钥走 .env（已 gitignore），代码/commit/PR 里不出现任何 token；含密钥的文件写完 grep '\*\*\*' 验证没被脱敏器污染
  4. OpenClaw /v1/chat/completions 的 token = operator 凭据，同机/内网传递，不进配置面板截图
