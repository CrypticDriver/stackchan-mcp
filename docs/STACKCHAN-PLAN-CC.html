🤖 StackChan 具身智能玩法路线图

整理: 狗蛋 · 2026-07-10 v2（全面重写）· 交付对象: Claude Code (Happy) · 仓库: CrypticDriver/stackchan-mcp

0. 愿景（一句话）

StackChan 不是"又一个语音音箱"，而是 AI 的物理化身：脑在外面（狗蛋/Claude），桌上这台只是身体。

传统 LLM 只能被动看喂给它的东西；有了身体它就能主动去看、主动提醒、用表情和动作表达状态——这个 agency 的转变才是具身智能的本质（引自 embodied-claude 项目理念）。

明确不要的: 设备内置的小智(xiaozhi)云 Agent 自己当脑子——那是"音箱模式"。小智链路仅作过渡，最终退役。

1. 硬件（大哥手上这台）

型号: M5Stack M5StackChan AI Desktop Robot Kit (ESP32-S3)，M5Stack 与社区共创的开源 AI 桌面机器人。
主机（CoreS3）: ESP32-S3 双核 240MHz，16MB Flash / 8MB PSRAM，WiFi 2.4G + BLE；2.0" 高强度玻璃盖板电容触控屏；0.3MP 摄像头（GC0308，320×240 JPEG）；接近传感器；九轴姿态传感器（加速度+陀螺仪+地磁）；microSD 卡槽；1W 扬声器 + 双麦克风。
机身: USB-C 供电/数据；电池 ~700mAh；两个带反馈舵机（水平 360° 无限旋转 / 竖直 90°；固件 API 坐标 pan ±128 / tilt 0–90）；2 排共 12 颗 RGB LED；红外发射+接收；顶部 3 区触控板；全功能 NFC；机身 I2C 地址 0x6F。
开发平台: UiFlow2 / Arduino IDE / PlatformIO / ESP-IDF。
当前固件: StackChan-UserDemo V1.4.3（2026-07-02，M5Burner），已配网在线（2.4G WiFi）。

2. 架构：脑体分离

【目标架构 — 具身模式】
  狗蛋 (OpenClaw, AWS EC2)
    ⇅ Tailscale (tailnet, 免公网暴露)
  家里常开小主机 bridge (树莓派/N100/Mac)
    · stackchan-mcp (Python, 13 工具 = 身体驱动层)
    · faster-whisper (耳: 本地 ASR)  · edge-tts (声)
    · 人脸检测 (外脑视觉, 参考 badger)
    · Happy 工作监控轮询器 (M2, 见 §4)
    ⇅ LAN HTTP (:80)
  StackChan 固件 = 纯身体: 说/听/动/看/表情/LED

【过渡链路 — 将退役】RETIRE
  StackChan(小智Agent) → xiaozhi.me 云 → MCP 适配器 → 狗蛋
  (现在能用"问一下狗蛋…"对话, M4 完成后关闭)
身体能力清单（固件 HTTP API 已有，不用动固件）: TTS 播放(TCP/UDP/HTTP PCM 流) · 录音(消费式读取) · 舵机 /move /nod /shake /home · 7 种表情 /face (calm/thinking/happy/sleepy/shy/smug/pouty) · 摄像头 /snapshot · 状态诊断。

3. M1 — 身体驱动地基（先做，其他一切的前提）

M1基于 stackchan-mcp fork 增强，全部只改 Python：
  1. ASR 可插拔 + faster-whisper: 解掉 Fish Audio key 硬依赖（audio_processing.py L281 / listening.py L27）。ASR_ENGINE=fish|whisper；whisper 用 small 模型 / zh / CPU int8 / beam_size=5。验收: 无任何云 key 时 listen 端到端可用
  2. MCP HTTP Bearer 鉴权: ASGI middleware 校验 Authorization: Bearer $STACKCHAN_MCP_TOKEN，远程接入前置条件。验收: 无/错 token 401
  3. edge-tts 声音可配: 加 STACKCHAN_EDGE_VOICE_ZH env（用 zh-CN-YunxiaNeural，与狗蛋微信语音一致）
  4. TTS 缓存: hash(文本+voice) → WAV LRU 缓存，口头禅零延迟
  5. PCM numpy 向量化: audio_processing.py L196-249 纯 Python 逐 sample 循环 → numpy ~10x（树莓派必须）。验收: 输出逐字节一致
部署形态: 家里常开小主机 + Tailscale；狗蛋侧写 skill 直接 HTTP 调（不必强走 MCP 协议）。

4. M2 — 工作具身化：盯 Happy 干活 ⭐大哥钦点

M2场景: StackChan = Happy 工程团队的物理监工。大哥的 CC 们在 https://d2nkikyt4i91kk.cloudfront.net（自托管 Happy）上干活，机器人把"活儿的状态"变成物理信号:
  • 需要审批/输入（session 卡在 permission 请求或提问）→ 转头朝大哥 + thinking 表情 + 说"老大，XX 项目的 CC 在等你审批"
  • 任务完成 → happy 表情 + 点头 + 播报一句结论摘要（≤20 字，口语化）
  • 任务报错/失败 → pouty 表情 + 摇头 + 播报错误要点
  • 长任务进行中 → 屏幕保持 thinking 表情 + LED 呼吸灯（不出声，不打扰）
  • 安静时段（可配 22:00-08:00）→ 只做表情不出声
实现要点:
  • bridge 侧写一个 happy-watcher 轮询器: 用 Happy server API（自托管，token 鉴权）定期拉 session 列表与最新消息状态（/v1/sessions、/v2/sessions 等，注意消息 payload 端到端加密——优先评估复用 happy 客户端 SDK 解密；如工程量大，退化方案 = 只用 session 的 active/updatedAt/agentState 等明文状态字段做状态机）
  • 状态机去重: 同一 session 同一状态只提醒一次；批量变更合并成一句话
  • 提醒模板全部口语化短句，禁 markdown/长句（语音场景）
参考范本: PheelaV/stackchan-badger（"Gooseberry"）——coding agent 需要 approve 时机器人提醒，M5Stack 社区 2026-05 发布，本地 bridge + 事件广播，可抄思路。

5. M3 — 感知主动化：会看的机器人

M3
  • 人脸追踪转头: 固件低帧率推 /snapshot JPEG → bridge 跑人脸检测（OpenCV/mediapipe，算力在外脑）→ 回传归一化坐标 → 舵机跟随。badger 已验证此模式（macOS Vision 版），咱换 Linux 实现
  • 回家打招呼: 检测到人脸出现（且 N 分钟内首次）→ 转头 + happy + "老大回来啦"
  • 狗蛋主动看: 狗蛋远程调 snapshot 看家里情况（配合已有 image 分析能力），主动感知而非被动等指令
  • 接近传感器: 有人靠近唤醒屏幕/表情，离开进 sleepy

6. M4 — 对话直连（小智退役）

M4
  • 入口: 顶部触控板轻拍 或 wake word（本地 VAD+关键词，参考 robo8080 AI StackChan 系列）→ 录音 → bridge whisper 转写 → 直发狗蛋(OpenClaw API) → 回复 edge-tts 播出
  • 目标延迟 ≤2s；进阶可研究 realtime 双向 PCM 流（社区已有 CoreS3 + gpt-realtime PoC，server VAD，延迟 <1s——架构可借鉴，模型换掉）
  • 验收后关闭小智链路: 停用 xiaozhi-mcp-adapter systemd 服务 + 设备解绑小智 Agent

7. M5 — 生活流（日本社区成熟玩法移植）

M5早晨定时播报（天气/日程/隔夜要闻，狗蛋早报的语音版）· 定时提醒物理化（开庭/会议提前提醒=转头+说话）· 表情习惯（久无互动进 sleepy，被摸头 shy）· IR 遥控家电（机身自带红外发射，可学习空调/电视码）。

9. 安全红线（违反=返工）

  1. 固件 HTTP :80 全裸无鉴权（含摄像头 /snapshot 与麦克风 /audio——隐私面）: 绝不做路由器端口转发；远程一律 Tailscale
  2. MCP HTTP 加 token 之前不开任何公网 tunnel
  3. 密钥全走 .env（gitignored）；代码/commit 里不出现 token
  4. Happy server token / OpenClaw API 凭据只在 tailnet 内传递
  5. 摄像头/录音数据默认不出家庭网络；狗蛋远程调用需 tailnet + 日志留痕

10. 交付顺序与验收

  1. M1: fork 分支 feat/embodiment-m1，每项带单测（token 401/200、numpy 逐字节一致、whisper 中文样例），README 补环境变量说明
  2. M2: happy-watcher 独立模块 + 状态机测试；先在 fake 模式（打印代替真机）验证事件流，再接真机
  3. M3 → M4 → M5 按序推进；M4 完成并稳定一周后退役小智链路
  4. 每个 M 完成给大哥现场演示一次；演示脚本写进 docs/
